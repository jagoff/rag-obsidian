# Eval delta — auto-tuning de pesos del ranker (2026-04-15)

Snapshot del impacto de `rag tune` sobre el ranking final. Baseline corresponde
a los pesos hardcodeados históricos (`recency_cue=0.1, tag_literal=0, recency_always=0,
feedback_pos=0.03, feedback_neg=0.15`); winner son los pesos resultantes del
random search de 500 trials + coordinate-descent refinement, sembrado con
`--seed 42`.

## Contexto

- **Commit**: master (post-feat: `rag archive` y previo a `rag tune` ship).
- **Comando**: `rag eval` contra `queries.yaml` (21 singles + 25 chain turns / 9 chains).
- **Golden aumentado para tune**: 0 eventos de feedback con `corrective_path`
  (feedback.jsonl tiene 1 solo evento sin corrective → no aporta al golden).
  El sweep trabajó con 46 cases puros de `queries.yaml`.
- **Modelos**: `bge-m3` (embed), `BAAI/bge-reranker-v2-m3` (MPS+fp16),
  `qwen2.5:3b` (reformulation helper), `command-r:latest` (chat, no se usa en eval).
- **Objetivo**: `hit@5 + 0.5 * MRR` (combinado singles + chain turns).

## Pesos

| signal           | antes | después   |
|------------------|-------|-----------|
| recency_cue      | 0.100 | 0.301     |
| recency_always   | 0.000 | 0.055     |
| tag_literal      | 0.000 | 0.010     |
| feedback_pos     | 0.030 | 0.150     |
| feedback_neg     | 0.150 | 0.418     |

Lecturas:

- `recency_always` y `tag_literal` arrancaron en 0 y el sweep los subió —
  son señales nuevas que el retrieval nunca había aprovechado. La mejora
  de singles viene de ahí.
- `feedback_pos/neg` se movieron a la parte alta del rango. El RAG tenía
  la mecánica de feedback desde hace tiempo pero con deltas muy
  conservadores; el sweep aprendió a confiar más en las señales del
  usuario (justificable: sobre queries con feedback existente, ese signal
  era casi el único que discriminaba).
- `recency_cue` subió de 0.10 a 0.30 — notas-con-cue-temporal ganan 3×
  más boost. Consistente con que el gate ya existía, sólo estaba
  sub-ponderado.

## Cómo reproducir

```bash
# 1. Baseline — borrar cualquier config previa del ranker.
rm -f ~/.local/share/obsidian-rag/ranker.json
rag eval | grep "Singles\|Chains:"

# 2. Tune (reproducible via --seed).
rag tune --samples 500 --seed 42 --apply

# 3. Re-eval con pesos persistidos.
rag eval | grep "Singles\|Chains:"
```

## Cuándo re-correr

- Cambios en chunking, embedder, reranker o helper LLM → re-tune
  (los features cambian de escala, los pesos óptimos también).
- Crecimiento del feedback log (>10 eventos con corrective_path) →
  re-tune aprovecha más señal.
- Cada ~mes como rutina — se logea todo en
  `~/.local/share/obsidian-rag/tune.jsonl` y sólo se persiste si gana.
